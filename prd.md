# BiteRhythm (食之律动) - 产品需求文档 (PRD)

本项目旨在解决家庭在工作日“决定吃什么”的日常痛点。通过微信小程序，结合大模型（LLM）智能规划与搭配，辅助家庭管理员（如雇主）快速制定健康、多样且不重复的周菜单，并实现与家庭成员、做饭阿姨的多端数据共享与权限管控。

---

## 1. 用户痛点与核心目标

* **用户痛点**：
  * **决策疲劳**：每周都需要为工作日（周一至周五）想好5天的菜谱，决策过程费时费脑。
  * **口味与重复性问题**：需要频繁变换花样，保证荤素搭配、营养均衡，且避免短时间内菜品重复。
  * **多方信息不对称**：雇主规划的菜单需要方便地共享给做饭阿姨（只读查看）和家人（共同决策或查看），目前缺乏一个统一、实时的共享看板。
* **核心目标**：
  * 提供一个基于**日历**的直观菜单规划工具。
  * 利用 **大模型（LLM）** 结合用户收藏的菜品库和历史饮食记录，智能推荐/一键生成菜单，解决决策疲劳。
  * 基于 **微信小程序云开发** 实现免服务器维护的数据存储与实时共享。
  * 提供轻量级的**权限管理系统**，确保雇主作为超级管理员拥有绝对控制权，阿姨仅能只读查看，家人可根据需要授予读写权限。

## 2. 核心功能模块设计

### 2.1 家庭资料管理 (Family Management)
用户可以创建和管理一个或多个“家庭”单元，因为不同家庭（或不同时期的家庭组合）有不同的饮食偏好。
* **多家庭管理**：支持新增、编辑、删除家庭。
* **状态持久化**：用户可随时切换当前家庭，且小程序会记录用户最后一次选中的家庭，在下次进入时自动加载。

### 2.2 日历与每日菜谱 (Calendar & Daily Menu)
以月历形式直观展示每一天的菜单状态。
* **月历视图**：可自由左右滑动或点击按钮切换月份，展示当月所有日期。
* **日期单元格状态**：日历单元格内简要展示当天已规划的菜品名称（如“小炒肉+西红柿炒蛋”），未规划的日期留白或显示待规划标识。
* **详情编辑页**：点击任意日期单元格，进入该日的菜谱编辑页：
  * 展示当天日期、星期。
  * **今日菜谱（不分餐段）**：只规划一顿饭（如晚餐），提供统一的“今日菜品”列表。
  * 支持**手动添加菜品**：支持直接输入文本新增，或点击“从我的收藏/菜品库中选择”。
  * 支持**一键生成/推荐**：调用大模型，基于家庭偏好和本周其他日期的菜谱，自动推荐今日菜品。
  * 支持**清空、复制**等快捷操作（如一键复制昨天的菜单）。

### 2.3 我的收藏与菜品库 (Dishes Repository)
作为一个专属的家庭“菜谱备忘录”，是手动点菜和大模型推荐的基础数据源。
* **菜品列表**：展示所有已收藏的菜品，支持按标签（如：肉类、素菜、汤、主食；或川菜、粤菜等口味）分类浏览。
* **菜品管理**：支持快速新增菜品（输入菜名、类型、主料、备注）、编辑和删除。
* **快速导入**：支持通过大模型解析一段话，自动提取并批量导入菜品。

### 2.4 大模型助手 (LLM Assistant)
这是本小程序的核心差异化亮点，用于解决“头疼吃什么”的问题。
* **系统配置与设置中心（限超级管理员）**：
  * **全功能锁定保护**：为保障家庭隐私与模型密钥安全，整个系统“设置”页面仅限超级管理员（雇主）可访问查看。其他成员（家人、阿姨）访问时将显示锁定保护界面。
  * **二级子菜单结构**：
    1. **成员与协作管理**：超级管理员在此子页面查看当前成员列表，通过微信生成邀请链接，审批待加入的新成员申请并授予其相应权限。
    2. **大模型参数配置**：超级管理员配置 API 提供商（如 DeepSeek, OpenAI, 腾讯混元等）、API Key 和 API Base URL，配置安全保存在云开发数据库中，不在前端暴露。
* **一键智能排餐（周/日）**：
  * 用户可以选定下周的5个工作日，点击“大模型智能排餐”。
  * LLM 结合以下维度生成菜单：
    1. **收藏的菜品**：优先从用户收藏的菜品库中挑选。
    2. **去重规则**：避免相邻几天出现完全相同的食材或菜品。
    3. **搭配原则**：自动实现“一荤一素”、“两荤一素一汤”等营养均衡的搭配。
    4. **家庭偏好**：避开忌口食材。
  * 排餐完成后，用户可对不满意的一天或某道菜进行“换一换”微调。
* **今日搭配建议**：在单日编辑页，LLM 根据当前的菜品，给出配菜建议。

### 2.5 多端共享与权限管理 (Permissions & Sharing)
通过微信 `OPENID` 实现安全的分布式协作。
* **超级管理员 (Super Admin) 锁定与设置保护**：
  * 系统的超级管理员通过绑定您的微信 `OPENID` 来唯一锁定。
  * 只有超级管理员可查看和操作系统设置，包含家庭成员改名、调权、新成员审批，以及大模型 API 配置。
* **成员管理与协作流**：
  * **微信分享邀请**：超级管理员可以在设置子页面生成家庭邀请，并发送给家人或阿姨。
  * **新用户填名加入**：新成员（家人、阿姨）首次点击邀请加入时，系统会提示其必须填入自己的真实姓名/昵称，提交加入申请。此后该成员无法在前端自行修改姓名，如需修改必须联系超级管理员在设置中进行调整。
  * **管理员统一审批与调权**：超级管理员在“设置 -> 成员与协作管理”中会收到待审批列表，审批通过并为其指定角色：
    * **读写权限 (Write/Read)**：适合普通家庭成员（家人），可以编辑每日菜单、添加/修改收藏菜品，但无权访问“设置”页面。
    * **只读权限 (Read Only)**：适合做饭阿姨，只能在首页查看已规划的菜谱，无任何编辑和写入权限，且无权访问“设置”页面。

## 3. 技术方案概要

本项目将完全基于**无服务器架构（Serverless）**，利用微信小程序自带的**云开发 (CloudBase)** 能力：
1. **云数据库 (Cloud Database)**：使用 JSON 文档型数据库，建立五张核心表：
   * `system_config`（系统配置表）：存放 LLM 提供商、API Key 等。仅超级管理员的 OpenID 可读写。
   * `families`（家庭表）：存放家庭资料（名称、偏好、就餐人数）。
   * `family_members`（家庭成员及权限表）：存放 `family_id`、`openid`、`nickname`、`role` 等信息。
   * `dishes`（菜品收藏表）：存放各家庭收藏的菜品。
   * `menus`（菜单计划表）：以日期和 `family_id` 为唯一索引，存放每天的菜谱。
2. **云函数 (Cloud Functions)**：
   - `auth`：微信登录及获取 OpenID。
   - `adminService`：管理员专属服务（保存 LLM 配置、审核成员等），执行时校验请求者 OpenID 是否为硬编码的超级管理员。
   - `memberService`：处理申请加入、查看当前家庭成员列表等。
   - `menuService`：增删改查菜单及菜品库，执行时校验请求者在当前家庭的权限等级（只读权限拦截写入）。
   - `llmService`：在云函数端安全调用大模型 API 并返回推荐结果。
3. **安全规则 (Security Rules)**：通过云数据库的安全规则，限制只读用户对 `system_config` 和 `menus`/`dishes` 的直接修改，多重保障。

## 4. 已确认的产品决策 (Aligned Decisions)

以下为在需求分析阶段与超级管理员（开发者）已确认的核心产品设计决策，将作为开发阶段的约束条件：

1. **系统设置全功能锁定**：系统设置下的所有功能（成员管理与大模型配置）均为超级管理员（雇主）专属，对普通家庭成员及阿姨完全锁定并显示安全保护锁屏。
2. **设置菜单两级结构**：设置页面内拆分为“成员与协作管理”和“大模型配置”两个子菜单，大模型配置安全存储于微信云数据库中。
3. **成员改名及调权由超管独占**：新用户在被邀请接受加入时须填写姓名，后续该姓名仅能由超级管理员在设置页面内修改，超级管理员同时独占对成员角色的升级/降级调整权限。
3. **超级管理员身份锁定**：本小程序的超级管理员为唯一指定用户（即您本人），开发时将通过配置文件或云函数初始化时硬编码绑定您的微信 `OPENID`，以确保绝对的系统管理安全性。
4. **菜单餐段规划**：每日菜单不区分“午餐”和“晚餐”餐段，仅作为一个统一的“今日菜品”列表进行排餐和展示，简化交互流程。
